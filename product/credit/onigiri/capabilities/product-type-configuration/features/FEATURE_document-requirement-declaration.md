# Feature: Document Requirement Declaration

**Parent Capability**: Product Type Configuration — [CAPABILITY](../CAPABILITY.md)
**Product**: Onigiri — [PRODUCT](../../../PRODUCT.md)
**Engineering Owner**: TBD
**Status**: Concept
**Last Updated**: 2026-03-18

---

## User Story

As a **Product Owner**, I want to declare which evidence is required per section and configure upload boxes for each evidence entry, so that Credit Officers see the correct document upload interface and Matcha receives the right document checklist — without engineering deployment.

## Job-to-be-Done

Today, the `document_verification_mapping` table is seeded via code. This feature replaces that with an Admin UI where POs declare evidence requirements per section in the Product Type Builder's accordion cards. POs configure what customers must bring, how many upload slots each evidence has, and under what conditions evidence is required.

---

## 3-Level Document Hierarchy

This feature operates at the **Evidence** and **Upload Box** levels of Onigiri's 3-level document model:

| Level | What It Is | Owner | Configured Where |
|-------|-----------|-------|-----------------|
| **Document Type** | AI classification category — tells Matcha what kind of document it's looking at | **Engineering** | Document Type Registry ([FEATURE](FEATURE_document-type-registration.md)) |
| **Evidence** ★ | What the customer must bring — a named requirement tied to a section | **PO** | Product Type Builder → section accordion card |
| **Upload Box** ★ | A specific upload slot within an evidence — what the CO actually uploads | **PO** | Per evidence, within section accordion card |

### Key Relationships

- **One document type → many evidence entries.** Example: document type `copy_of_id_card` is used by both "สำเนาบัตรประชาชนผู้กู้" (Customer ID Card) and "สำเนาบัตรประชาชนผู้ค้ำ" (Guarantor ID Card). Same AI classification, different evidence contexts.
- **One evidence → many upload boxes.** Example: evidence "Car Registration Book" has upload boxes: หน้าปก, หน้ากรรมสิทธิ์ล่าสุด, หน้ากรรมสิทธิ์ก่อนหน้า, หน้าภาษี, หน้าบันทึกเจ้าหน้าที่ 18 เป็นต้นไป, รูปถ่ายเอกสารหลักประกันกับลูกค้า.
- **Evidence is per section.** Each evidence entry belongs to a specific section in the product type (configured in the section's accordion card).

### Two Evidence Categories

| Category | Description | Configured By | Examples |
|----------|------------|---------------|---------|
| **Customer / Guarantor Documents** | Physical documents the customer or guarantor must bring to CO | **PO** per section | ID Card (customer), ID Card (guarantor), Registration Book, Insurance, Income Docs |
| **System-Generated Documents** | Documents generated by the system during the application process | **System** (same for all product types) | Contract, PDPA Consent Form, Insurance Application Form |

System-generated documents are **always the same across all product types** — PO does not configure them. They are auto-included in every product type's document checklist.

---

## Scope

**IN scope:**
- Declare evidence requirements per section in the Product Type Builder accordion cards
- Configure upload boxes per evidence (name, order, allow multiple, min/max files, tooltip)
- Link each evidence entry to a document type (for AI classification)
- Configure conditional inclusion/exclusion rules per evidence
- Select a Data Extraction Template per evidence (from engineering-provided templates)
- Auto-include system-generated documents (not editable by PO)
- Preview the resulting document checklist (showing which evidence + upload boxes would appear for a given set of field values)
- Auto-populate `document_verification_mapping` rows at product type activation

**OUT of scope:**
- Matcha verification rule configuration (Matcha owns how it verifies a document)
- Document type creation (Engineering-owned — see [Document Type Registration](FEATURE_document-type-registration.md))
- System-generated document set management (product-level decisions, not per-product-type)

## Evidence Configuration UI

### Evidence Declaration (per section accordion card)

PO adds evidence entries within each section's accordion card in the Product Type Builder:

| Config Field | Required | Description |
|-------------|----------|-------------|
| Evidence name | Yes | Display name for the evidence (e.g., "สำเนาบัตรประชาชนผู้กู้") |
| Evidence name (EN) | No | English name (e.g., "Customer ID Card Copy") |
| Document type | Yes | Selected from Document Type Registry — for AI classification (e.g., `copy_of_id_card`) |
| Required | Yes | Always / Conditional |
| Conditional rule | If conditional | `WHEN [field] = [value] THEN EXCLUDE` |
| Data extraction template | Yes (default available) | Engineering-created template mapping application fields to Matcha `check_name` keys |

### Upload Box Configuration (per evidence)

Each evidence entry has one or more upload boxes — the actual upload slots the CO sees:

| Config Field | Required | Default | Description |
|-------------|----------|---------|-------------|
| Upload box name | Yes | — | Display name for the upload slot (e.g., "หน้าปก", "หน้ากรรมสิทธิ์ล่าสุด") |
| Order in upload group | Yes | Auto-increment | Display order within this evidence's upload boxes |
| Allow multiple upload | Yes | `false` | Whether CO can upload more than one file to this slot |
| Minimum files | Yes | `1` | Minimum number of files required (0 = optional slot) |
| Maximum files | Yes | `1` | Maximum number of files allowed (only relevant when Allow Multiple = true) |
| Tooltip message | No | — | Help text shown to CO on hover |
| Tooltip image | No | — | Example image shown to CO (e.g., photo showing which page to photograph) |

```

├── Evidence: สำเนาบัตรประชาชนผู้กู้ (Customer ID Card Copy)
│   Document Type: copy_of_id_card       ← same AI classification
│   Upload Boxes: [สำเนาหน้า-หลัง]
│
├── Evidence: สำเนาบัตรประชาชนผู้ค้ำประกัน (Guarantor ID Card Copy)
```

Both evidence entries use the same document type `copy_of_id_card` for AI classification, but they are separate evidence requirements with separate upload boxes.

### Conditional Inclusion Rules

Same pattern as before, but applied at the **evidence** level (not document type level):

```
WHEN [field_name] = [value] THEN [INCLUDE|EXCLUDE] [evidence_name]
```

| Constraint | Rule |
|-----------|------|
| Trigger field | Must be a field in the evidence's parent section |
| Condition | Equality only (`= value`) |
| Action | Include (default) or Exclude |
| Multiple conditions per evidence | Not supported — one rule per evidence. If complex logic needed, escalate to engineering. |

**Example (Collateral: Bike section):**
- Evidence "สมุดทะเบียนรถจักรยานยนต์" — Always required
- Evidence "กรมธรรม์ประกันภัย" — Always required
- Evidence "ใบตรวจสอบจาก DLT" — `WHEN bike_act_type = "RY-17" THEN EXCLUDE`

### Mapping to `document_verification_mapping` Table

At product type activation, each evidence entry is compiled into `document_verification_mapping` rows:

| Source (Admin UI) | Target (DB Column) |
|------------------|-------------------|
| Evidence name | `evidence_name` |
| Document type key | `document_type_key` |
| Section (from product type) | `section_key` |
| Conditional rule | `conditional_expr` (compiled to JMESPath expression) |
| Data extraction template | `extraction_template_key` (FK to `extraction_template` table) |
| Upload boxes (array) | `upload_boxes` (JSON array with name, order, allow_multiple, min, max, tooltip_message, tooltip_image) |

### Data Extraction Templates

A **Data Extraction Template** is a pre-configured set of mappings from application field paths to Matcha `check_name` keys. It tells Onigiri Worker which values to extract from the application and include in the Matcha task payload.

**Ownership:** Engineering creates and maintains templates. This requires understanding both Matcha's `DocumentVerificationItem` configuration (which `check_name` keys exist per document type) and Smart Form's field paths (which application fields exist).

**Template Schema:**

| Field | Type | Description |
|-------|------|-------------|
| `template_key` | String | Unique identifier (e.g., `thai_id_card_standard`) |
| `document_type_key` | String | Which document type this template applies to |
| `mappings[]` | Array | One entry per Matcha data item |
| `mappings[].check_name` | String | Must match a Matcha `DocumentVerificationItem.check_name` |
| `mappings[].field_path` | String | Application field path referencing Smart Form field keys (e.g., `identity.id_national_id`) |
| `mappings[].transform` | String (optional) | Transformation expression (e.g., concatenation: `identity.id_first_name + " " + identity.id_last_name`) |

**Example — `thai_id_card_standard`:**

| check_name | field_path | transform |
|-----------|-----------|-----------|
| `id_card_number` | `identity.id_national_id` | — |
| `full_name_th` | `identity.id_first_name`, `identity.id_last_name` | concatenate with space |
| `date_of_birth` | `identity.id_date_of_birth` | — |

**PO selects a template** from the available templates for the chosen document type. Most document types have one default template. If multiple templates exist (e.g., for different section variant combinations), PO picks the appropriate one.

## Acceptance Criteria

| # | Criterion | Pass Condition |
|---|-----------|---------------|
| AC-1 | PO adds an evidence requirement to a section | Evidence entry appears in the section's accordion card with document type, name, and required status |
| AC-2 | PO configures upload boxes for an evidence | Upload boxes created with name, order, allow multiple, min/max files; displayed in correct order |
| AC-3 | PO configures tooltip for an upload box | Tooltip message and/or image saved; shown in preview |
| AC-4 | PO configures a conditional rule on evidence | Rule saved; preview shows evidence excluded when condition met |
| AC-5 | PO selects data extraction template | Template key stored in mapping row; available to Onigiri Worker at runtime |
| AC-6 | Same document type used for multiple evidence entries | System allows selecting the same document type for different evidence entries (e.g., customer ID + guarantor ID) |
| AC-7 | Activation populates mapping table | `document_verification_mapping` rows created matching declared evidence + upload boxes |
| AC-8 | System-generated documents auto-included | Contract, PDPA, Insurance Form included without PO action; shown as read-only in preview |
| AC-9 | Preview shows effective checklist | PO enters sample field values; preview displays which evidence + upload boxes would appear |
| AC-10 | Document type must exist in registry | System prevents referencing an unregistered document type |
| AC-11 | Conditional rule references valid field | System validates trigger field exists in the evidence's parent section |
| AC-12 | Version immutability | Activating a new version replaces mapping rows for that product type; old version's rows archived |

## Edge Cases & Error States

| Scenario | Expected Behavior |
|----------|------------------|
| PO removes an evidence entry | All upload boxes for that evidence are also removed; PO warned |
| PO adds evidence with 0 upload boxes | Validation error: "At least one upload box required per evidence" |
| Section field referenced in condition is deleted | Validation error at save: "Field [X] no longer exists in section" |
| Product type activated with 0 section-specific evidence | Allowed — only system-generated documents sent to Matcha |
| Extraction template references a field path that doesn't exist in the selected variant | Validation warning at configuration time; runtime: Worker logs warning, sends null |
| Re-activation (new version) | Old mapping rows soft-deleted; new rows inserted; in-flight applications use pinned version |
| Upload box max < min | Validation error: "Maximum files cannot be less than minimum files" |
| Upload box min = 0 | Allowed — this makes the upload box optional within a required evidence |

| Dependency | Type | Notes |
|-----------|------|-------|
| Document Type Registration | Internal — [FEATURE](FEATURE_document-type-registration.md) | Document types must exist in registry before they can be referenced in evidence declarations |

| `document_verification_mapping` table | Internal — Onigiri DB | Target table for compiled evidence + upload box configurations |
| Onigiri Worker | Internal | Worker reads mapping table + extraction template at `create_facility` to build Matcha POST /task payload |
